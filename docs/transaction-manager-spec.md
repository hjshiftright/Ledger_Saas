# Transaction Manager — Detailed Design Document
## Ledger 3.0 — Module 2: Transaction Manager
### Version 0.1 | Date: March 15, 2026

---

## Design Philosophy

The Transaction Manager is the operational heart of Ledger. Every rupee that enters or leaves any
account — whether imported from a bank statement, triggered by a mutual fund SIP, manually
entered after a cash purchase, or automatically generated by a recurring rule — is processed
here. If the Chart of Accounts is the skeleton of the user's financial life, the Transaction
Manager is the bloodstream: the constant, ongoing flow that keeps everything alive and current.

Three design principles govern this module.

**Principle 1 — The ledger is sacred, but the UI should not feel like accounting.** The
double-entry engine beneath the surface is uncompromising: every transaction must balance, every
correction must be an audit-trailed reversal, every journal entry line must have an account and
an amount. But the user never sees the words "debit," "credit," or "journal entry." They see
"moved ₹5,000 from HDFC savings to ICICI savings" or "categorized as Dining Out." The accounting
rigour is invisible infrastructure.

**Principle 2 — Import should feel like magic, not plumbing.** The user uploads a PDF.
They should experience: it worked, here are my transactions, do these look right? The complexity
of password decryption, table extraction, source detection, FIFO matching, and deduplication
happens silently. The only thing asked of the user is a light review and a final confirmation.
A user who has never heard of a "hash-based deduplication engine" should still feel perfectly
comfortable importing statements.

**Principle 3 — Nothing is final until the user says so, and nothing is lost ever.** Before a
transaction becomes a journal entry, the user gets a chance to review and correct it. After a
transaction is confirmed, it can still be corrected — through a clean edit workflow that creates
a reversal under the hood. And every import can be undone. The user should feel in control at
every step and should never fear that they have permanently broken something.

## Module Overview & Navigation

The Transaction Manager is not a single screen. It is a cluster of nine tightly related sub-modules, each with a distinct job, all sharing the same ledger database underneath. The user reaches this module from the main navigation sidebar under **Transactions**, which expands into the following areas:

**Import** brings the user to the Import Hub — the place to upload bank statements, CAS PDFs, and broker exports. This is where raw data enters the system.

**Review Queue** is where imported and auto-generated transactions wait for the user's confirmation. Nothing becomes a permanent ledger entry until it passes through here. A red badge on this item shows the count of pending items at all times — it is the module's conscience, a constant gentle reminder that unreviewed data is not yet real.

**All Transactions** is the Transaction List — the permanent ledger view of everything that has been confirmed. This is where the user browses their financial history, searches for specific transactions, and, when needed, edits or removes entries.

**New Transaction** opens the Manual Entry screen — for recording cash expenses, income, account transfers, investment purchases, and any financial event that does not come from an imported statement.

**Recurring** is the Recurring Rules screen — a register of all scheduled transactions: SIPs, EMIs, rent, subscriptions, insurance premiums.

**Rules** is the Categorization Rules Manager — the user's view into how the system maps bank narrations to spending categories, with the ability to add, edit, and override rules.

**Import History** is the full audit log of every document ever uploaded, with the ability to undo or re-process any past import.

**Reconcile** is the Statement Reconciliation tool — for verifying that the Ledger's balances match the user's official bank and investment statements to the rupee.

On mobile, these areas collapse into a bottom navigation bar with icons only. The most frequently used surfaces — Review Queue badge, New Transaction shortcut, and Transaction List — are always one tap away.

## The Contextual Helper Panel

A contextual helper panel is present throughout the Transaction Manager, occupying the right 30% of the screen on desktop. The helper panel observes what the user is currently doing — uploading a file, reviewing a flagged transaction, looking at an EMI entry — and proactively surfaces relevant guidance. When the user has configured an LLM API key in Settings, the helper panel is fully conversational. Before the API key is configured, it surfaces curated, context-aware tips in a chat-like format.

Examples of what the helper panel says at each stage are shown throughout this document in italics.

On mobile, the helper panel collapses to a floating question-mark button that opens a bottom sheet.

## Sub-module 1: Import Hub

**Purpose:** Bring external financial data into Ledger by uploading bank statements, mutual fund CAS PDFs, broker exports, and any other supported financial document.

**Primary persona:** Every user, from their first week. The Import Hub is typically the second thing a user visits after completing onboarding — the first thing is the dashboard, which prompts them with a nudge to import their first statement. Returning users visit the Import Hub once a month, after receiving their bank statement by email or downloading it from net banking.

**UX Concept — The Filing Cabinet That Does the Work For You**

Most people have a folder on their laptop called something like "Bank Statements" — a graveyard of PDFs that were dutifully downloaded and never opened. The Import Hub is the antidote. The user drops a file. The system identifies it, decrypts it, extracts every transaction, checks for duplicates, and categorises everything. The user's job is simply to drop the file and confirm the results. The complexity is completely invisible.

The page is designed around instant feedback. Every action — file selected, source detected, parsing complete — produces a visible response within two seconds. Waiting with no feedback is the enemy of trust. The user should never wonder whether anything is happening.

### 1.1 Hub Layout

The Import Hub is a dedicated page accessible from the main navigation sidebar under
"Transactions > Import." It is also reachable from the post-onboarding call-to-action card that
nudges the user to import their first statement.

The page is divided into two panels.

The **left panel** (approximately 40% width) is the upload area. It contains a prominent
drag-and-drop zone with the text "Drop your statement here, or click to browse" and accepts
PDF, CSV, XLS, and XLSX files. Below the drop zone, a "Supported Formats" expandable section
lists every supported source with a downloadable guide for obtaining each statement type (e.g.,
"How to download your HDFC bank statement," "How to export your Zerodha Tradebook"). This
reduces the most common support question: where to find the statement.

The **right panel** (approximately 60% width) is the active import status panel. It shows the
current upload + parse status for the most recently uploaded file, and below it, a compact
import history showing the last 5 imports with their status. A "View All Imports" link leads
to the full Import History sub-module (Part 5).

### 1.2 Upload Flow — Step by Step

**Step 1 — File selection:**
The user drags a file onto the drop zone or clicks to open the file browser. Multiple files can
be selected in one go — they will be queued and processed sequentially. Each file appears as a
row in an "Upload Queue" list below the drop zone with a status indicator (Queued, Processing,
Done, Error).

File validation happens immediately on selection:
- Reject files larger than 50 MB with: "This file is too large. Bank statements should be under
  50 MB. Try splitting into smaller ranges if needed."
- Reject non-PDF/CSV/XLS/XLSX files with a clear error.
- For PDFs: if the file is encrypted, proceed to Step 2. If not encrypted, skip to Step 3.

**Step 2 — Password entry (for encrypted PDFs only):**
A modal appears showing the detected source (e.g., "This looks like an HDFC Bank statement")
and a password hint box: *"For HDFC statements, the password is your PAN in uppercase followed by your date of birth in DDMMYYYY format. Example: ABCDE1234F01011985."*

The password field is a standard password input with characters masked and a "Show password" toggle. The user types the password and clicks **Open Document**. A spinner confirms decryption is in progress — client-side only. If decryption succeeds, the modal closes and parsing begins automatically. If it fails, the hint refreshes with an alternative format suggestion: *"Alternatively, some HDFC accounts use just the date of birth in DDMMYYYY format. Give that a try."* The user gets two more attempts before a **Can't open this file?** help link appears, which expands a troubleshooting guide listing all known password format conventions for the detected bank.

The password decryption step is designed to feel like unlocking a door, not navigating a security obstacle. The hint system means most users succeed on the first attempt.

> **Security note:** The password is processed and discarded entirely on the client side — in the browser's memory or in the native app's local process. It is never sent to any server, never written to disk, and never logged. The Ledger backend receives only the decrypted document content.

**Step 3 — Source detection and parsing:**
The file is analysed to identify its source. A spinner shows in the upload queue row with the
label "Identifying document…". Within a few seconds this updates to the detected source:
"HDFC Bank Statement (PDF)" or "CAMS Consolidated Account Statement" or "Zerodha Tradebook
(CSV)."

If the source cannot be identified with high confidence, the label reads "Unknown format —
your help needed" and the user is taken to the column-mapper UI (see §1.4).

For large files (over 10 MB or over 500 pages), parsing is moved to a background job and the
user sees: "This is a large file — parsing in the background. We'll notify you when it's ready."
The user can navigate away. An in-app notification (bell icon in the header) fires when
parsing completes.

**Step 4 — Parsing progress:**
For normal-sized files parsed synchronously, a real-time progress indicator shows:
- "Extracting text from 12 pages…"
- "Found 143 transactions…"
- "Checking for duplicates…"
- "Categorizing transactions…"
- "Done — 98 new transactions ready for review."

**Step 5 — Transition to Review Queue:**
On completion, a success card appears showing the deduplication summary:
> "143 transactions found. **98 are new.** 45 were already in Ledger (skipped). 3 transfer
> pairs detected and linked. 12 transactions need your attention."

A single "Review Transactions →" button takes the user directly to the Review Queue
pre-filtered to this import batch.

The helper panel during the upload flow surfaces tips based on the detected source. When an HDFC statement is detected: *"HDFC statements cover one calendar month. If you have statements from multiple months, you can upload them all at once — just select them all in the file picker. They'll be queued and processed in order, and duplicates across statement periods will be caught automatically."* When a CAMS CAS is detected: *"Your CAMS Consolidated Account Statement covers all your mutual fund folios linked to this PAN, across all AMCs. A single CAS file is usually enough to capture your entire mutual fund history."*

### 1.3 Account Mapping

When a statement is imported for the first time for an account that does not yet exist in the
user's Chart of Accounts, the system prompts the user before parsing begins:

> "We found a statement for an ICICI Savings account (XXXX4521). Should we create this account
> in your Ledger?"  
> **[Create Account]** **[Map to Existing Account]** **[Cancel]**

If "Map to Existing Account" is selected, a dropdown of existing bank accounts is shown. This
is useful when the user has an account already set up under a different name.

If "Create Account" is selected, the account is created mid-flow under `Asset > Bank > ICICI
Savings (XXXX4521)` and the import proceeds. The user can rename the account later from the
Chart of Accounts settings.

### 1.4 Column Mapper UI (Generic / Unrecognized Files)

When the parser cannot confidently identify a CSV or Excel file's column structure, the
Column Mapper UI is shown. This is a non-blocking interactive screen, not a modal —
the user can take their time.

The screen shows a scrollable preview table of the file's first 10 rows. Above the table,
a row of **semantic slot cards** is displayed: Date, Narration, Debit, Credit, Balance
(and optionally: Reference Number, Value Date). Each card is a drop target.

The column headers from the file are shown as draggable chips. AI-suggested mappings are
pre-applied as a starting point — the chips are already placed in their most likely slots,
with a subtle glow to indicate they were auto-placed. The user adjusts as needed.

Once all required slots (Date, Narration, and at least one of Debit/Credit) are filled, a
"Validate" button becomes active. Clicking it runs a quick balance cross-check and shows the
result: "Balance check passed — ready to import" or "Balance check failed — total debits and
credits don't reconcile with the opening/closing balance. Try remapping the amount columns."

When the mapping is confirmed, it is saved as a named profile that the system will reuse if
the same file format is uploaded again: "Saved mapping for 'Yes Bank CSV Export'. We'll
remember this for next time."

### 1.5 Opening Balance Handling on First Import

When a statement is the first import for a given account, the system checks whether the statement contains a running balance column. If it does, the system works backwards from the closing balance and all transactions to compute what the opening balance must have been. This is displayed to the user for confirmation:

*"We calculated an opening balance of ₹42,300 for your HDFC Savings account as of 01-Apr-2025, the start of this statement. Does this look right?"*

Two options are presented: **Confirm ₹42,300** and **Enter a different amount**. Most users will confirm. Some users will enter a different amount if they know their actual balance was different — perhaps because they had an outstanding cheque that hadn't cleared on that date.

If the statement does not contain a running balance column (some CSV exports omit it), the user is asked to enter the opening balance directly. The helper panel explains why this is needed: *"The opening balance is the amount in this account just before the first transaction in this statement. It makes sure Ledger's running balance matches your actual account balance. You can find this number by checking your bank's app or net banking portal for the account balance on that date."*

Once confirmed, the opening balance is recorded as a journal entry dated at the statement start date, with the counterpart to an Opening Balance Equity account. This keeps the accounting equation in balance from day one, even before any expenses or income are recorded. The user never sees this entry in their normal transaction list — it is tucked away under a "System Entries" filter and is only visible when explicitly requested.

The helper panel adds a tip after the opening balance is set: *"This is a one-time step for each account. Future imports for this account will pick up exactly where the last one left off — no opening balance needed again."

## Sub-module 2: Review Queue

**Purpose:** Let the user review, classify, and confirm every imported or auto-generated transaction before it permanently enters the ledger.

**Primary persona:** Every user, immediately after importing a statement. The Review Queue is the "inbox" of the Transaction Manager. A well-designed queue session should feel like quickly sorting mail — fast, satisfying, and done in a few minutes.

**UX Concept — Inbox Zero for Your Finances**

The Review Queue is inspired by the experience of clearing an email inbox — but designed for the financial world. Each transaction is an item that needs to be acknowledged: the AI has done its best to categorise it, and the user's job is to confirm that categorisation or improve it. Most items should require no more than a glance. A small number will need more thought. A very small number will reveal something unexpected.

The queue is designed around two tempos. The first is the **bulk tempo**: the user scans the high-confidence green rows, confirms the categorisations look right at a glance, and clicks "Approve All High Confidence" to dispatch them in one action. The second is the **careful tempo**: the user slows down for yellow and red rows — flagged items, low-confidence categorisations, newly discovered transfer pairs — and gives each one the attention it deserves. The interface never forces the careful tempo onto the bulk-tempo items or vice versa.

The Review Queue is not optional. It is a deliberate gate. The philosophy is: if you haven't reviewed it, it isn't real yet. This protects users from silent errors — a bank narration that the AI misread, a transfer that got categorised as an expense — that would otherwise silently corrupt their financial data.

### 2.1 Queue Layout

The Review Queue is accessible at "Transactions > Review Queue." A persistent badge on the
navigation item shows the count of pending items (e.g., "Review Queue (98)").

The page is split into two sections:

The **left section** (65% width) is the transaction table. Each row represents one parsed
transaction. Columns: Checkbox | Date | Narration | Account | Suggested Category | Amount
(colour-coded: green for credits, red for debits) | Confidence | Flags.

The **right section** (35% width) is the detail pane. When a row is selected, the detail
pane shows the full transaction information and editing controls. The detail pane is sticky —
it stays visible as the user scrolls the queue table.

### 2.2 Row Priority Colouring

| Row Colour | Condition | Meaning |
|---|---|---|
| Green | Confidence > 0.90, no flags | High confidence — safe to bulk-approve |
| Yellow | Confidence 0.60 – 0.90, or minor flags | Needs a quick glance |
| Red | Confidence < 0.60, or critical flags | Must review individually |
| Blue | Transfer pair (two-account transfer) | System detected as an inter-account transfer |
| Purple | Recurring transaction (auto-generated) | Confirm this month's recurring entry |

### 2.3 Filtering and Sorting the Queue

Above the table, a filter bar provides:
- **Import batch filter** — "Showing: HDFC Statement Apr 2025 (98 items)" with a dropdown
  to switch between batches or show "All pending"
- **Status filter** — All / Needs Review / High Confidence / Transfer Pairs / Flagged
- **Date range filter**
- **Account filter** — filter to a specific source account

A sort control allows ordering by: Date (default), Amount, Confidence (ascending — shows
most uncertain items first), and Flags.

### 2.4 Bulk Actions

A "Select All" checkbox at the top of the table selects all currently visible rows. Three
bulk action buttons appear when any rows are selected:

**"Approve Selected"** — Approves all selected rows using their current suggested categories.
All selected rows must be green or yellow — red rows cannot be bulk-approved (they require
individual attention).

**"Set Category for Selected"** — Opens a category picker. The chosen category is applied to
all selected rows. Useful for batches of similar transactions (e.g., selecting all Swiggy
transactions and confirming "Dining Out" in one action).

**"Mark as Transfer"** — Marks selected rows as an inter-account transfer (see §2.7).

At the top of the queue, a prominent "Approve All High Confidence" button approves all green
rows in one click. A confirmation popover shows: "This will approve 67 transactions. Continue?"

### 2.5 Detail Pane — Transaction Review

When the user clicks any row, the right-side detail pane populates with:

**Transaction header:** Date, narration (full, untruncated), source account, amount, and the
import batch it came from.

**Category assignment:** A searchable dropdown pre-filled with the AI-suggested category.
The dropdown shows the full account hierarchy (e.g., `Expense > Food & Dining > Dining Out`)
and supports typing to filter. Frequently used categories for this account are shown at the
top of the dropdown. A "Not sure? Ask AI" link opens a small chat bubble that accepts a
free-text question and returns a categorization suggestion.

**Journal entry preview:** A compact, clean display showing what accounting entries will be created when this transaction is approved. For a Swiggy food order of ₹842 debited from HDFC Savings:

```
  will record as:
  Food & Dining › Dining Out       ₹842   ←  money spent here
  HDFC Savings                     ₹842   ←  money came from here
```

This is deliberately not labelled "Debit" and "Credit." The language is plain: money was spent here, it came from there. But structurally, this is a double-entry journal entry — the user is seeing real accounting without knowing they're doing accounting. For users who are curious, a small **What does this mean?** link expands a one-paragraph explanation of double-entry bookkeeping in everyday language: *"Every transaction affects two places in your finances. Here, your Dining Out spending went up (money was spent), and your HDFC Savings balance went down (money left). These two effects always mirror each other — that's what keeps your financial picture accurate."*

For multi-leg transactions — a salary credit that splits into gross pay, TDS, and EPF — all legs are shown. The helper panel explains the complexity: *"Your salary credit has been split into three parts based on your salary structure: the gross amount, the TDS deducted, and your EPF contribution. You can adjust these splits if the amounts look different from your actual payslip."*

This preview is the most important financial education element in the application. Every time a user reviews this panel, they understand their finances slightly better — without a single lesson.

**Notes field:** A free-text field for the user to add a note (e.g., "Team lunch — Prasad's
farewell"). Notes are searchable in the Transaction List.

**Tags field:** A comma-separated tag field (e.g., "team, work, reimbursable"). Tags support
filtering and custom reports later.

**Split transaction button:** See §2.6.

**Actions:** "Approve" (green button), "Skip for Now" (saves the review state and moves to
next row), "Exclude" (marks transaction as intentionally ignored — it won't appear in any
reports and won't be approved).

### 2.6 Split Transactions

The Split Transaction feature handles a common scenario: one bank debit needs to be divided
into multiple categories. For example, a ₹12,000 Amazon order that includes ₹8,000 of
electronics (asset purchase) and ₹4,000 of household supplies (expense).

Clicking "Split" on a transaction opens an inline split editor within the detail pane:

The original transaction amount (₹12,000) is shown at the top. Below it, a list of split
lines appears, each with: a category picker and an amount field. Initially, one split line
is pre-populated with the full amount and the AI-suggested category.

The user clicks "Add Split Line" to add more lines and adjusts the amounts. A running
total below the list shows: "Allocated: ₹12,000 / Remaining: ₹0" — it turns red if the
amounts don't sum to the original. The "Approve" button is disabled until the split
amounts exactly equal the original transaction amount.

The resulting journal entry for the ₹12,000 Amazon example:

Dr: Asset > Electronics ₹ 8,000
Dr: Expense > Household > Supplies ₹ 4,000
Cr: Asset > Bank > HDFC Savings ₹12,000

Splits created during review are preserved — if the user later re-opens the transaction
from the Transaction List, they see the split breakdown and can edit it.

### 2.7 Transfer Detection and Linking

One of the most common sources of error in personal finance tracking is treating internal account transfers as income or expenses. When ₹20,000 moves from the user's HDFC Savings to their ICICI Savings, both statements show a transaction — but neither is income and neither is an expense. It is a pure movement of money between two pockets.

The deduplication engine (SM-F, R2.6) handles this automatically. Match criteria for a valid transfer pair:

- **Opposite direction** — one row is a debit, the other is a credit
- **Equal amount** — to the rupee (no fuzzy matching; bank transfers settle exactly)
- **Date proximity** — same calendar day or ±1 day (to handle NEFT cut-off lag)
- **Transfer narration signal** — at least one of the cleaned narration or the original bank narration contains a transfer keyword: `upi`, `neft`, `rtgs`, `imps`, `transfer`, or `trf`

#### Scenario 1 — Same-batch detection (both accounts uploaded together)

The user uploads Account A's statement and Account B's statement in one go. Both end up in the same import batch. During deduplication, the engine performs a pairwise scan across all new rows — when it finds a matching debit/credit pair, both rows are flagged `TRANSFER_PAIR` immediately. Both appear in the Review Queue highlighted in blue with a linked-chain icon.

The detail pane shows both sides together:

*From: HDFC Savings — ₹20,000 on 14 Mar 2026*
*To: ICICI Savings — ₹20,000 on 14 Mar 2026*

Approving either row approves both simultaneously. A single inter-account transfer journal entry is created — no income account, no expense account. Neither transaction appears on spending reports.

#### Scenario 2 — Cross-batch detection (accounts added at different times)

The user added Account A months ago and imported its history. They now add Account B. During the new import, the engine compares Account B's incoming rows against Account A's historical rows (passed as `existing_rows` to the dedup service). When a match is found:

- The **new** row (Account B) is flagged `TRANSFER_PAIR` in the Review Queue — it appears in blue and shows the linked historical row
- The **historical** row (Account A) is retroactively updated with `dedup_status = TRANSFER_PAIR` and `transfer_pair_retroactive = True` — it surfaces in the Review Queue as a **Retroactive Link** requiring a brief confirmation

The helper panel explains: *"We found that this ₹20,000 in your new ICICI statement matches a transfer from your HDFC account that was already in Ledger. We've linked them — your historical entry has been updated so it no longer shows as an unexplained debit."*

If the historical row had already been approved (generating a journal entry with a Suspense debit), the system flags the existing journal entry for correction. The user sees: *"This previously approved transaction now has a match. Confirm to update the journal entry — the Suspense account will be replaced with ICICI Savings as the counterpart."*

#### Scenario 3 — One-sided import (counterpart account not yet added)

The user imports Account A but hasn't added Account B yet. The NEFT debit in Account A has no matching credit to pair with. The row stays `NEW` and is categorised and proposed normally (with a Suspense/Unknown counterpart account in the journal entry). No transfer detection fires.

When Account B is added later and its statement is imported, Scenario 2 takes over: the engine retroactively links the historical Account A row and prompts the user to correct the previously approved journal entry.

**This means users should not worry about the order of account imports.** The system catches transfers whenever both sides become available, whether that is in the same batch, the next day, or six months later.

#### Scenario 4 — Bank to brokerage (not a simple transfer)

A NEFT debit from Account A to Zerodha, followed by a stock purchase in the Zerodha tradebook, is **not** a transfer pair. These are two distinct financial events that happen to be related:

1. Account A debit → `DR ZerodhaLedger / CR AccountA` (fund movement)
2. Zerodha BUY trade → `DR Holdings:INFY / CR ZerodhaLedger` (securities purchase)

The dedup engine does not attempt to link a bank debit to a brokerage trade. Each generates its own journal entry. The Zerodha ledger account acts as the intermediary that ties both entries together in the overall balance sheet.

#### UX — After pairing

Both rows appear in the queue with a **Transfer Pair** badge. Clicking either opens a transfer detail pane:

*From: HDFC Savings — ₹20,000 on 14 Mar 2026*
*To: ICICI Savings — ₹20,000 on 14 Mar 2026*
*Matched via: NEFT narration signal*

Approving approves both simultaneously. A **Break Link** button is available if the pairing is incorrect — dissolving it routes each transaction independently through the categorisation flow.

The helper panel explains: *"We've detected that this ₹20,000 transaction is a transfer between two of your own accounts. Approving it here means it won't show up as an expense on your spending reports — which is correct, since you haven't actually spent anything."*

### 2.8 Approval → Journal Entry Creation

When a transaction is approved (individually or in bulk):

1. The double-entry engine validates that debit total equals credit total. If not, the
   approval is rejected with an error — this should never happen for system-generated entries
   but acts as a safety net for manually edited ones.
2. A permanent `JournalEntry` record is created with a sequential entry number
   (e.g., `JE-2026-001234`), the entry date, all JE legs (account, debit/credit, amount,
   quantity if applicable), the source transaction ID, and the import batch ID.
3. The account running balance is updated for every affected account.
4. The transaction row disappears from the Review Queue.
5. The Reporting module's cache is invalidated for all affected accounts and the covering
   date period.

This flow is irreversible from the queue — once approved, corrections go through the
Transaction List edit flow (see Part 3).

## Sub-module 3: Transaction List — The Ledger View

**Purpose:** Browse, search, and manage all confirmed financial transactions in a single view. Also serves as the passbook view for individual accounts.

**Primary persona:** The user who wants to find something specific ("when did I last pay my LIC premium?"), verify their spending in a category, or review a particular account's history. This is the module the user consults most often after the dashboard.

**UX Concept — The Passbook Reimagined**

Every Indian who has held a bank account knows the passbook — a physical booklet with every transaction in chronological order, a running balance after each entry, and the reassuring permanence of ink on paper. The Transaction List is the digital descendant of that passbook, but dramatically more powerful: searchable, filterable, multi-account, organised by category, with a full journal entry visible behind every row.

The design challenge is that personal finance transaction lists are simultaneously boring (most individual rows are routine and unremarkable) and critical (the one non-routine row you're looking for might be buried 400 rows deep). The answer is a powerful, fast search-and-filter system paired with a clean default view that doesn't overwhelm. When nothing is searched or filtered, the list shows recent transactions in a simple, readable format. When the user needs to dig, every tool is available — and the results appear instantly.

### 3.1 Page Structure

The Transaction List page is accessible from "Transactions > All Transactions."

A persistent **scope selector** at the top of the page controls the view:
- **All Accounts** (default) — all transactions across all accounts, sorted by date descending
- **Single Account** — a dropdown to select one account and view its ledger (equivalent to
  a bank passbook for that specific account)

When a single account is selected, a running balance column appears showing the account's
balance after each transaction — this is the passbook view (R4.6).

### 3.2 Filters and Search

A prominent search bar at the top accepts free-text search against: narration (full-text),
merchant/payee name, notes, and tags. Results are highlighted inline.

Below the search bar, a collapsible filter panel offers:

| Filter | Options |
|---|---|
| Date Range | This Month / Last Month / This Quarter / This FY / Last FY / Last 12 Months / Custom |
| Account | Multi-select from all accounts in the CoA |
| Category | Multi-select from all Category/Expense/Income accounts |
| Amount | Min amount / Max amount range |
| Transaction Type | All / Income / Expense / Transfer / Investment / Loan EMI |
| Tags | Free-text tag filter |
| Flags | Has note / Has attachment / Manually entered / Imported / Recurring / Needs Review |
| Import Batch | Filter to all transactions from a specific import |

Active filters are shown as dismissible chips below the filter panel. A "Clear All Filters"
link removes all active filters at once.

### 3.3 Transaction Row Design

Each row in the transaction list contains:

- **Date** — transaction date (not value date; value date shown in detail view if different)
- **Narration** — merchant name or description, truncated to ~60 characters with a "..." 
  expand on hover. If the user has set a custom label on this transaction, the custom label
  appears instead of the raw narration, with the original narration shown in a smaller grey
  sub-text.
- **Category** — the account path of the non-source leg, shown as a breadcrumb:
  `Food & Dining › Dining Out`
- **Account** — the source account (for single-account view, this column is hidden)
- **Amount** — right-aligned, colour-coded green for credits, red for debits
- **Running Balance** — shown only in single-account view; the account balance after this
  transaction
- **Icons** — small icon indicators for: has note (speech bubble), has attachment (paperclip),
  is recurring (repeat icon), was manually entered (pencil icon), is part of a split
  (scissors icon)

### 3.4 Transaction Detail View

Clicking any row opens a side panel (or a full modal on mobile) showing the complete transaction
detail:

**Header section:** Full narration, date, value date (if different), import source (e.g.,
"Imported from HDFC Statement — Apr 2025") or "Manually entered."

**Journal Entry section:** A clean, readable presentation of all legs:

Journal Entry JE-2026-001234 · 15 Apr 2025

Dr Expense › Food & Dining › Dining Out ₹842.00
Cr Asset › Bank › HDFC Savings ₹842.00

For multi-leg entries (salary, EMI split), all legs are shown. An "Accounting view" toggle
switches the presentation to a traditional debit/credit table format for users who prefer it.

**Metadata section:** Notes, tags, import batch reference, original bank narration (if the
user has set a custom label), transaction hash (for debugging, shown under an "Advanced" toggle).

**Attachments section:** The user can attach files to any transaction — a photo of a receipt,
a warranty card, an invoice. Attachments are stored in the Document Vault (Module 12.2 in the
PRD). A maximum of 5 files per transaction, 10 MB per file.

**Action buttons:** Edit, Duplicate, and Delete (which creates a reversal).

### 3.5 Editing a Confirmed Transaction — The Immutable Ledger

**Design philosophy:** Once a transaction is confirmed, it cannot be silently altered. This is not a limitation — it is the foundation of a trustworthy ledger. In double-entry bookkeeping, the correct way to fix an error is to reverse it and record the correction. This preserves the complete history: what was originally recorded, when it was corrected, and what it was corrected to. A ledger that allows silent edits is a ledger that cannot be audited or trusted.

For the user, this is completely invisible. They click "Edit," make their change, click "Save" — and the system handles the reversal transparently.

Per R3.6, confirmed transactions cannot be silently edited — corrections are made through
reversing entries. The UI abstracts this completely.

When the user clicks "Edit" on a confirmed transaction, the detail pane opens in edit mode.
They see the same fields as the review queue detail pane: category picker, split lines, notes,
tags. The original values are pre-populated.

When they click "Save Changes," the system silently:
1. Creates a reversing entry dated today:


// Reversal of JE-2026-001234
Dr Asset › Bank › HDFC Savings ₹842.00 (reversal)
Cr Expense › Food & Dining › Dining Out ₹842.00 (reversal)

2. Creates a new correcting entry with the updated categorization:

// Correction
Dr Expense › Shopping › Grocery ₹842.00
Cr Asset › Bank › HDFC Savings ₹842.00


The user sees none of this complexity. From their perspective, the transaction simply has
a new category. However, the original entry and the reversal are both preserved in the audit
trail and visible if the user clicks "View Edit History" on the transaction.

A small "Edited" badge appears on the transaction row in the list to indicate it has been
corrected from its originally imported value.

### 3.6 Deleting a Transaction

"Delete" is labelled "Remove from Ledger" to signal its permanence. A confirmation dialog
appears: "This will remove this transaction and reverse its accounting entries. This action
cannot be undone."

Behind the scenes, the system creates a reversing entry (same as §3.5 Step 1) and marks
the original entry as deleted. The transaction no longer appears in the transaction list
(it is filtered out by default) but remains archived in the audit log, viewable from
Import History.

## Sub-module 4: Manual Transaction Entry

**Purpose:** Record any financial transaction that does not come from an imported statement — cash expenses, income, account transfers, investment purchases, opening balance adjustments.

**Primary persona:** The user who just paid for an auto-rickshaw in cash, received a gift from a relative, or needs to log a freelance payment received in a WhatsApp bank transfer that isn't yet reflected in a statement. Manual entry is a daily-driver feature on mobile and an occasional-use feature on desktop.

**UX Concept — Frictionless at the Moment of Transaction**

The key insight about manual entry is that the moment of highest motivation is when the transaction just happened — the user is still holding the receipt, still watching the UPI notification slide down from the top of their phone screen. The entry must be completable in under fifteen seconds, while standing in a queue or walking out of a shop. The Quick Entry flow is designed for exactly this: three decisive taps and it's done.

The Full Entry flow exists for the other end of the spectrum — the user who sits down on a Sunday evening to record a week's worth of cash expenses, or needs to log a complex multi-part salary entry. Both flows are available from the same entry point: the floating **+** button.

### 4.1 Quick Entry (Simplified Mode)

Quick Entry is designed for the most common case: a cash or card expense recorded on the
spot. It is accessible from a floating "+" button that is visible throughout the application
(not just in the Transaction Manager), and as a keyboard shortcut (Ctrl+N / Cmd+N).

The Quick Entry panel slides up from the bottom (mobile) or appears as a compact popover
(desktop). It has just four fields:

**Amount** — numeric input, large font, accepts typing naturally (e.g., "842" not "₹842.00").

**Category** — a searchable category picker showing frequently used expense accounts at the
top. Typing "swig" brings up "Dining Out" immediately. The user does not need to navigate the
full account hierarchy.

**Date** — defaults to today. A compact date selector allows changing it.

**Note** — optional free text field for a brief description.

Tapping "Add" creates the transaction immediately. The journal entry is generated:

Dr [Selected Expense Account] [Amount]
Cr Asset > Cash in Hand [Amount]

The default credit account is Cash in Hand. If the user wants to record a card expense,
a small "Paid with" toggle at the bottom of the Quick Entry panel allows switching the
credit account to any bank or credit card account.

### 4.2 Full Entry (Advanced Mode)

For transactions that require more control — income entries, investment purchases, multi-leg
salary records — the Full Entry screen is available. It is accessible from
"Transactions > New Transaction" in the navigation, or by clicking "Switch to Full Entry" in
the Quick Entry panel.

The Full Entry screen presents a structured form:

**Transaction date** — date picker, defaults to today.

**Description** — free-text narration field. The AI categorization engine analyses this in
real-time and suggests a category as the user types. Typing "SBI home loan EMI" triggers
the EMI-splitter and pre-populates the entry legs with the calculated principal/interest split.

**Journal entry legs** — a table where the user sees the accounting entries directly:

| Account | Debit | Credit |
|---|---|---|
| Expense > Housing > Loan Interest | ₹14,240 | — |
| Liability > Loans > SBI Home Loan | ₹22,760 | — |
| Asset > Bank > SBI Savings | — | ₹37,000 |
| Total | ₹37,000 | ₹37,000 ✓ |

The user can add, remove, and edit rows. The total line shows debit and credit totals in
real-time, with a green ✓ when they balance and a red ✗ when they don't. The "Save" button
is disabled until the entry balances.

**AI Assist button:** A small "Suggest Entries" button sends the description to the AI
and asks it to suggest a complete balanced journal entry. The suggestion populates the legs
table, which the user can then adjust. This is particularly helpful for complex transactions
like salary credits (gross pay, TDS, EPF, net amount).

**Notes and tags** — same as the Review Queue detail pane.

**Recurring toggle** — "Make this a recurring transaction" — see Part 5.

### 4.3 Income Entry

Recording income follows the Full Entry flow. The most common income transaction —
salary credit — is given a shortcut:

A "Record Salary" quick-action (available from the Quick Entry panel as a secondary option)
opens a pre-structured salary entry form. If the user has previously set up a salary structure
(gross pay components, TDS rate, EPF contribution), all fields are pre-populated from the
last recorded salary. The user only needs to confirm or adjust amounts and click "Record."

The resulting journal entry for a salary credit:

Dr Asset > Bank > HDFC Savings ₹85,000 (net in-hand)
Dr Asset > Provident Fund > EPF ₹ 4,500 (employee PF contribution)
Dr Expense > Tax > TDS ₹10,500 (tax deducted at source)
Cr Income > Salary ₹1,00,000 (gross salary)


### 4.4 Transfer Between Accounts

Moving money between two of the user's own accounts — such as transferring from savings to
an FD, or topping up a wallet — is handled through a dedicated "Transfer" entry type
accessible from the Quick Entry panel.

The Transfer form has three fields: From Account, To Account, and Amount. Notes and date are
optional. The journal entry created is a pure balance sheet transfer:

Dr Asset > Bank > ICICI Savings ₹50,000
Cr Asset > Bank > HDFC Savings ₹50,000

No income or expense account is involved. The transaction appears in both accounts' ledger
views with the narration "[Transfer to/from ICICI Savings]."

### 4.5 Voice Input (Mobile)

On the mobile app, the Quick Entry panel includes a microphone button that activates voice
input. The speech-to-text works in English and Hindi. Examples of phrases the system can parse:

- "Spent 250 on auto rickshaw" → ₹250, Expense > Transportation > Auto/Cab
- "Paid 1200 electricity bill" → ₹1,200, Expense > Utilities > Electricity
- "Received 500 from Rahul" → ₹500, Income > Other Income (transfer or loan repayment
  — flagged for manual category confirmation since "Rahul" is not a known payee)

Voice input populates the Quick Entry fields, which the user confirms visually before saving.
The system does not blind-save voice-parsed entries — user confirmation is always required.

## Sub-module 5: Recurring Transactions

**Purpose:** Schedule predictable financial events so they are generated automatically and placed in the review queue on their due date, requiring only a quick confirmation rather than manual re-entry every cycle.

**Primary persona:** Every user with at least one of: a monthly rent payment, an SIP, an insurance premium, an EMI, or a subscription. In practice, this is almost every user.

**UX Concept — Set It Once, Confirm It Monthly**

The design question for recurring transactions is: should they be auto-approved, or should the user confirm each occurrence? Auto-approval feels convenient — the user sets up rent once and never thinks about it again. But it creates a silent failure mode: when the rent increases, or the SIP amount changes, or a subscription is cancelled, auto-approved amounts continue to record the old values until the user notices the discrepancy.

Ledger's answer is **default to confirmation, with opt-in auto-approval**. Each recurring item generates an entry in the Review Queue's purple section — a lightweight confirmation prompt that costs five seconds. For amounts that genuinely never change (a fixed-rate EMI, a specific subscription), the user can toggle **Auto-approve this item** on the rule, and future occurrences will be approved without review. A monthly digest notification lists all auto-approved items so the user maintains awareness without effort.

This means a user who has set up twelve recurring rules will see perhaps three or four in the purple queue each month for the ones they've chosen to confirm, and will receive a brief email listing the nine that were auto-approved. The combination of convenience and awareness is the goal.

### 5.1 Setting Up a Recurring Rule

A recurring rule can be created from two places:

**From Full Entry:** After composing a transaction, the user toggles "Make this recurring"
at the bottom of the form. A secondary options panel expands with: Frequency (Weekly, Monthly,
Quarterly, Half-Yearly, Annually), Day of month (e.g., "5th of every month"), End condition
("Never" / "After N occurrences" / "Until date"), and an optional label override.

**From the Recurring Rules screen** (Transactions > Recurring): A standalone form for
setting up rules from scratch, with the same fields as above.

When a recurring rule is saved, the system creates the first upcoming scheduled entry and
adds it to the Recurring Confirmation Queue.

### 5.2 Recurring Confirmation Queue

Ledger does not automatically approve recurring transactions — it puts them in a lightweight
Confirmation Queue. This is an intentional design choice: even predictable expenses occasionally
change (rent increases, SIP amount changes, EMI fluctuation due to rate changes), and
auto-approving amounts that may be wrong is worse than a quick monthly confirmation.

The Confirmation Queue appears as a dedicated section within the Review Queue (top of the
queue, above imported transactions), styled in purple (see §2.2). Each row shows: Transaction
name, scheduled date, amount, and category.

For amounts that are truly fixed (rent, subscription, SIP at a fixed amount), the user can
toggle "Auto-approve this recurring item" — future occurrences of this rule are approved
automatically without needing to enter the queue. A monthly digest email lists all
auto-approved recurring items for awareness.

### 5.3 Managing Rules

The Recurring Rules screen (Transactions > Recurring) shows all active rules in a table:
Rule name, category, amount, frequency, next due date, and status (Active / Paused / Ended).

Actions:
- **Edit** — modify the amount, category, or frequency
- **Pause** — stops generating new entries until manually resumed
- **End** — marks the rule as ended (no more new entries); historical entries are preserved
- **Record Manual Override** — records a one-off entry for this rule with a different amount,
  without changing the rule's recurring amount (useful for a month where the rent was
  different due to a partial month)

## Sub-module 6: Categorization Rules Manager

**Purpose:** Give the user full visibility and control over how the AI categorisation engine maps bank narrations to spending categories. Make the AI's behaviour transparent and correctable.

**Primary persona:** The power user who wants to understand why a transaction was categorised incorrectly, or who wants to set up a rule that correctly handles a narration pattern the AI doesn't recognise. This is not a screen most users visit frequently — they interact with it indirectly through corrections in the Review Queue. But when they do visit, they should find it clear and empowering.

**UX Concept — Make the Rules Visible, Make Them Yours**

AI categorisation is a black box for most users. They don't know why "NACH DR HDFCLIFE" was categorised as Insurance, or why "PAYMENT TO ZOMATO" was categorised as Miscellaneous instead of Dining Out. The Categorization Rules Manager opens the black box. Every categorisation decision traces back to a rule — either a system-built rule or a user-created rule. The user can see all of them, understand their logic, and change them.

The key design principle here is **transparency without overwhelm**. There are approximately 150 system rules. The user does not need to read all of them — they need to be able to find the rule that's causing a problem, fix it, and leave. The search bar and filter system makes this fast. The "User rules" tab surfaces only the rules the user has created or modified, which is typically a much smaller and more personally relevant list.

### 6.1 Rules List

The rules list shows all active categorization rules in a table:

| Column | Description |
|---|---|
| Rule Name | Auto-generated from narration pattern (e.g., "SWIGGY → Dining Out") |
| Narration Pattern | The string or regex pattern matched against transaction narrations |
| Assigned Category | The account the matching transaction is mapped to |
| Source | "System" (built-in) or "User" (created from a correction) |
| Match Count | How many transactions this rule has matched (all time) |
| Status | Active / Disabled |

A search bar filters the list. Clicking any row opens an edit panel.

### 6.2 System Rules vs. User Rules

**System rules** (labelled "Built-in") are shipped with Ledger — approximately 150 rules
covering common Indian merchants, UPI patterns, bank narration formats, and financial
infrastructure narrations (SIPs, EMIs, ATM withdrawals). System rules can be disabled but
not deleted. They apply to all users.

**User rules** are created automatically when the user corrects a categorization in the
Review Queue (after 3 identical corrections, a rule is auto-promoted). They can also be
created manually. User rules take priority over System rules.

### 6.3 Creating a Rule Manually

Clicking "Add Rule" opens a form with:

**Pattern type:** "Contains" (simple substring match — recommended for most users), "Starts
with," "Ends with," or "Regex" (power users only — a warning is shown: "Regex patterns can
match unintended transactions. Test carefully.").

**Pattern:** The text to match (e.g., "SWIGGY", "ZOMATO", "BESCOM"). Pattern matching is
case-insensitive.

**Assign to account:** A searchable dropdown of all accounts in the CoA.

**Apply to:** "New transactions only" (default) or "All existing matching transactions
(retroactive)" — selecting retroactive applies the rule to all historical transactions
in the Uncategorized or Miscellaneous category that match the pattern. A count is shown:
"This retroactive rule will re-categorize 47 existing transactions."

### 6.4 Rule Priority

Rules are evaluated in the following order:
1. User rules (exact match first, then contains)
2. System rules (exact match first, then contains)
3. AI categorization (for everything not matched by a rule)

If two rules match the same transaction, the more specific one wins (exact match over
substring, user over system). If there is a tie, the rule with the higher match count wins.

The user can manually set a priority number on their rules to override this default ordering.

## Sub-module 7: Import History

**Purpose:** Maintain a complete, auditable log of every document import, and provide the ability to undo or re-process any past import.

**Primary persona:** The user who suspects a statement was imported twice, or who wants to undo an import batch because they imported the wrong file. Also: the user who wants to confirm the last time they imported a particular account's statement.

**UX Concept — A Filing Cabinet With a Time Machine**

Import History is the receipt log for the import process. Every file ever uploaded has a record: when it was uploaded, what was found in it, how many transactions were new versus duplicates, whether parsing succeeded, and what the current status of the batch is. Nothing is lost.

The undo capability is the most important feature in this sub-module. The ability to fully reverse an import — including all the accounting entries it created — removes a large anxiety from the import process. The user can upload a statement, see something unexpected, and undo the entire import cleanly. This encourages the right behaviour: import often, reconcile regularly, and don't be afraid to try things.

### 7.1 History Table

A scrollable table with columns: Date Imported, Filename, Source Type, Statement Period,
Transactions Found / New / Duplicates, Parse Confidence, and Status.

Status values: Completed (all transactions approved), In Review (pending queue items
remain), Partially Reviewed, Failed, Rolled Back.

Each row is clickable and expands to show the full deduplication report for that import.

### 7.2 Undo an Import

Every completed import has an "Undo Import" action available. Clicking it triggers a
confirmation dialog:

> "This will reverse all journal entries created from this import batch
> (98 transactions). Your account balances will be updated accordingly.
> Transactions still in the Review Queue from this batch will be discarded."

Before allowing the undo, the system checks for blocking dependencies: if any subsequent import has a statement opening balance that was computed using the running balance established by this import, the undo would leave those accounts in an inconsistent state. If blocking dependencies are found, the system explains: *"This import cannot be undone because a later import for [Account Name] depends on its balances. Please undo the later import first."* The system lists the blocking imports with links so the user can undo them in the correct sequence.

If no dependencies exist, the undo proceeds silently. Counter-entries are created for every journal entry in the batch. All affected account balances are updated. The import batch is marked **Rolled Back** with the timestamp. The transactions are removed from the Transaction List's default view but remain accessible under the Import History detail — archived, not deleted. The user can always see what was in a rolled-back import and, if needed, re-import the file.

### 7.3 Re-Processing an Import

When the Ledger parsing engine is updated — for example, to handle a new HDFC statement format that changed in January, or to correctly split a new type of NACH narration — users who imported statements before the update can benefit from the improvement without finding and re-uploading their old files.

The **Re-Process** option on any historical import batch does the following: first, it rolls back the original import batch (creating reversals for all its journal entries). Then it re-runs the original stored file through the updated parser. The freshly parsed transactions land in the Review Queue as a new batch, ready for the user to review. The user sees the parser's improved categorisations and can approve them.

A notification is sent to users when a parser update affects source types they have previously imported: *"We've improved our HDFC Bank statement parser. Would you like to re-process your HDFC imports to benefit from more accurate categorisation?"*

## Sub-module 8: Statement Reconciliation

**Purpose:** Verify that the Ledger's balances for a given account exactly match the official statement from the bank or financial institution, catching any discrepancies due to missing transactions, duplicate entries, or data entry errors.

**Primary persona:** The methodical user — someone who wants to be certain their Ledger is a perfect mirror of reality, not just an approximation. Reconciliation is optional, but for users who use manual entry in addition to imports, it is the only way to catch errors that accumulate over time.

**UX Concept — Closing the Books**

In corporate accounting, reconciling a bank account is a routine month-end task: match every ledger entry against the bank statement, identify any outstanding items, and confirm the balances agree. For individuals, this level of rigour is unusual — but remarkably valuable. A reconciled account balance is a verified balance. An unreconciled balance is an estimate.

Ledger makes reconciliation accessible without making it mandatory. The Reconcile screen is always available from any account's ledger view, but it never nags. For users who want the additional assurance, it provides a clean, guided workflow. For users who are satisfied with imported data alone, they can simply ignore it.

The visual metaphor is the **running counter**: as the user marks transactions as cleared, a live counter shows how close the reconciled balance is to the official statement balance. The moment the counter reaches zero — the instant the two numbers match — the screen responds with a green confirmation. That moment of closure is the emotional reward for a reconciled ledger.

### 8.1 When Reconciliation Is Most Valuable

Reconciliation is most useful in three scenarios. The first is **mixed-method accounts** — accounts where the user both imports statements and enters some transactions manually (cash withdrawals that are then spent in cash, for example). Over time, discrepancies naturally accumulate between the two methods, and reconciliation catches them.

The second is **accounts with time gaps** — where the user imported statements covering January to March and then May to July, but not April. The April gap leaves a period where transactions are missing, and the account balance in Ledger may not reflect reality. Reconciliation at the end of July immediately reveals the discrepancy.

The third is **peace of mind** — for users who simply want to be certain that Ledger is accurate. Many of the same users who enjoy personal finance tracking are also the users who want to be sure. Reconciliation gives them certainty.

### 8.2 Starting a Reconciliation

Reconciliation is initiated from any account's ledger view. A **Reconcile This Account** button appears in the account header. Clicking it opens the Reconciliation screen for that account.

The screen begins with a single input: the closing balance from the user's official statement, and the date that balance corresponds to. *"Enter the closing balance from your HDFC Savings statement as of 31 March 2026."* The user types the amount and clicks **Start Reconciling**.

### 8.3 Marking Transactions as Cleared

The reconciliation screen shows every unreconciled transaction in Ledger for this account up to the target date, presented chronologically. Each row has a checkbox on the left.

The user marks each transaction as **Cleared** (it appeared on the official statement) by checking the checkbox. Transactions that are pending or outstanding — a recent UPI credit that hasn't settled yet, a cheque that hasn't been presented — are left unchecked.

As the user checks transactions, a sticky header at the top of the screen updates a live reconciliation panel:

*Statement Balance: ₹1,42,830*
*Cleared Transactions: 47 items totalling ₹1,38,450*
*Outstanding Transactions: 5 items totalling ₹4,380*
*Ledger Balance as of this date: ₹1,42,830*
*Difference: ₹0 ✓*

The **Difference** line is the key number. When it reaches zero, the account is reconciled. The counter is coloured red when non-zero and turns green the moment it reaches zero — providing instant visual feedback on progress.

### 8.4 Resolving Discrepancies

If the Difference does not reach zero after all expected transactions are marked, the system analyses what's left and surfaces likely explanations as contextual notes below the reconciliation panel.

*"There are 2 transactions in Ledger with no apparent match in your statement — these may be duplicate entries or manually entered transactions that were already captured in an import. Review them to see if they should be deleted."* Links are provided to the suspicious transactions.

*"Your statement balance is ₹500 higher than expected — this could mean there is a transaction on your statement that has not been imported or manually entered yet."* A shortcut to the Import Hub is provided.

The user resolves discrepancies by adding missing transactions, removing duplicates, or fixing incorrect amounts in the Transaction List. The reconciliation screen refreshes as changes are made, and the Difference counter updates in real time.

### 8.5 Completing the Reconciliation

When the Difference reaches zero, two things happen. First, the screen shows a completion card: *"Your HDFC Savings account is fully reconciled as of 31 March 2026. Every transaction in Ledger matches your official statement."* A **Complete Reconciliation** button is presented.

Clicking it locks the reconciliation period. Every cleared transaction receives a lock icon in the Transaction List — a visual indicator that this transaction has been verified against an official statement. Locked transactions cannot be edited or deleted without first unlocking the reconciliation period (which requires a deliberate confirmation and creates an audit note).

Future reconciliations for this account begin from the date of the last completed reconciliation, carrying forward any outstanding (unchecked) items from the previous session.

## Sub-module 9: Investment Account Specifics

**Purpose:** Provide additional tracking capabilities for quantity-based investment accounts — equities, mutual funds, ETFs, gold — where it is not enough to track money in and money out. The quantity held, the average cost per unit, and the current market value are all equally important.

**Primary persona:** The investor — any user who has stocks, mutual funds, or ETFs. This includes Persona 1 (Riya, who has Zerodha and Kuvera) and Persona 3 (Ananya, who has just opened a Zerodha account). Persona 2 (Suresh) also benefits for his SGBs and gold holdings.

**UX Concept — Your Portfolio Is More Than a Balance**

A bank account has one number: the balance. An investment account has many numbers: what you put in, what it is worth now, how many units you hold, what each unit cost you on average, and how much you have gained or lost both in absolute terms and as an annualised return. The standard transaction list cannot capture this richness. The investment account view adds a **Holdings** layer on top of the standard transaction layer — the same account, two lenses.

### 9.1 The Holdings Tab

Every investment account — whether created during onboarding, through an import, or manually — has a **Holdings** tab alongside its standard **Transactions** tab. The Holdings tab is the portfolio view for that specific account.

The Holdings tab shows a table with one row per security or fund currently held. The columns are: Security or Fund Name (with the AMC or exchange shown in smaller text below), ISIN, Quantity (current units or shares), Average Cost (the volume-weighted average purchase price per unit across all buy transactions), Cost Value (Quantity × Average Cost — what you paid for the current holding), Current Price (the latest known market price, with date), Market Value (Quantity × Current Price), Unrealised P&L (Market Value minus Cost Value, with percentage change), and XIRR (the annualised return accounting for the timing of every purchase, redemption, or SIP — the most accurate measure of actual performance).

The **XIRR column** deserves special attention. Unlike a simple percentage return, XIRR accounts for when each rupee was invested. A fund where ₹5,000 was invested in 2020 and ₹1,000/month has been added since then has a different true return than one where ₹17,000 was invested in a lump sum last year, even if the current portfolio value is the same. The helper panel explains this on first view: *"XIRR is your true annualised return — it accounts for when you invested each amount. It's more accurate than a simple percentage return, especially for SIP investments where money goes in over time."*

### 9.2 Lot-Level Expansion

Every row in the Holdings table expands to show the individual purchase lots that make up the current holding. This is the FIFO view — each lot shows when it was bought, at what price, the current value, the unrealised gain or loss for that lot, the holding period, and the tax classification (STCG for lots held less than 12 months, LTCG for lots held 12 months or more).

This lot-level view is the foundation for tax-loss harvesting. A user can look at their holdings, find lots that are at a loss, and decide to sell those specific lots to crystallise the loss for tax set-off purposes — while keeping appreciated lots to defer taxation. The helper panel supports this: *"You have 3 lots of HDFC Bank shares sitting at a loss totalling ₹8,400. If you sell and re-buy these lots, you can book this loss and use it to offset capital gains from other investments, reducing your tax liability. Consult your tax advisor before acting on this."*

### 9.3 Price Updates

Market values in the Holdings tab are only as current as the last price update. A **Update Prices** button at the top of the Holdings tab triggers a fresh price fetch from the configured market data integrations: AMFI for mutual fund NAVs, NSE or BSE data for equities. The last-updated timestamp is shown for each security: *"Price as of 14 Mar 2026, 3:30 PM IST."*

If market data APIs are not configured (e.g., the user is on a free tier or has not connected a data source), the user can manually enter the latest price for any security by clicking the Current Price cell directly. A small edit icon appears on hover. Manual price entries are tagged with a pencil icon to distinguish them from API-fetched prices.

For mutual fund NAVs specifically, the AMFI API is free and publicly available, so NAV updates are always available regardless of any paid integration tier. Equity prices require a market data connection.

### 9.4 Quantity-Tracked Transaction Entry

When a BUY transaction is entered manually for an investment account, the Full Entry form adds two fields that do not appear for non-investment accounts: **Quantity** (number of units or shares) and **Price per unit** (the price at which each unit was purchased). The Amount field auto-computes as Quantity × Price per unit and is confirmed by the user.

For SELL transactions, a **Lot Picker** appears after the user enters the sell quantity. The Lot Picker shows all available lots (from all prior purchases) with their dates, quantities, average cost, and current gain/loss. The system defaults to FIFO — the oldest lots are selected first — but the user can change the selection. Selecting different lots produces different capital gains amounts and different STCG/LTCG classifications. The system computes and displays the resulting capital gain or loss before the user confirms: *"Selling these lots will realise a Long-Term Capital Gain of ₹18,340. This is taxable at 12.5% above the ₹1.25 lakh annual exemption. Your total LTCG this financial year so far is ₹45,200."*

This pre-confirmation capital gains display is a small but powerful feature — it turns investment transactions into tax-aware decisions rather than blind clicks.

## Accounting Engine Guardrails

The Transaction Manager operates on top of the double-entry accounting engine. The following constraints are enforced at the engine level and cannot be bypassed from any part of the UI. They exist to ensure the integrity of the ledger.

**The balancing rule** is absolute. Every journal entry must have total debits equal to total credits. The engine rejects any entry that does not balance before writing it to the database. In practice, the UI enforces this visually long before the user attempts to save — the balance indicator turns green when the entry balances and red when it does not, and the Save button is disabled until the entry balances. The user can never inadvertently create an unbalanced entry.

**Immutability** is the foundation of auditability. Once an entry is confirmed, it cannot be silently edited. All changes happen through reversal and re-entry. The UI abstracts this — the user clicks Edit, makes their change, and clicks Save — but behind the scenes, the original entry is reversed and a correcting entry is created. Both the original and the correction are permanently visible in the audit trail. Even system administrators cannot erase historical journal entries — they can only add corrections.

**Account normal balances** are enforced with a warning (not a hard block). If a transaction would cause an asset account to carry a negative balance (e.g., a withdrawal from a bank account that exceeds the recorded balance), the system warns: *"This transaction will leave your HDFC Savings account with a balance of -₹2,400. This could mean there are transactions not yet recorded in Ledger, or that the opening balance was incorrect. Proceed anyway?"* The user can override the warning — negative balances are sometimes technically correct (an overdrawn account, a credit card account showing the amount owed) — but the warning ensures it is intentional rather than accidental.

**Period boundaries** prevent entries from being dated before an account's opening balance date. If the user tries to enter a transaction dated before the account's first entry, the system shows: *"The opening balance for this account is dated 01 April 2025. Transactions cannot be recorded before this date — the financial history for this account starts here."* To go further back, the user must update the account's opening balance date.

## Data Model Summary

*The following describes the key data entities and their relationships in plain terms. For the full technical schema, refer to the Engineering Design Document.*

The central entity is the **Journal Entry** — an immutable record with a sequential number (JE-2026-001234 format), a date, a narration, and two or more legs. Each leg references an account and carries either a debit amount or a credit amount (one of them is always zero). For investment accounts, legs also carry a quantity and a unit price. Journal entries cannot be deleted — they can only be reversed.

Before a journal entry exists, there is a **Pending Transaction** — the intermediate state that lives in the Review Queue. A Pending Transaction carries all the same information as a Journal Entry plus a suggested category, an AI confidence score, a set of flags, and a review status. It becomes a Journal Entry when it is approved. If it is excluded or skipped indefinitely, it remains as a Pending Transaction but is removed from the active queue view.

**Import Batches** group Pending Transactions together by their source file. Every file upload creates an Import Batch. The batch tracks parse confidence, deduplication results, and status. Undoing an import undoes the entire batch — not individual transactions.

**Recurring Rules** are templates for journal entries that should be generated on a repeating schedule. They carry a template entry, a frequency, a day-of-period setting, and a start/end condition. Each generated occurrence becomes a Pending Transaction in the purple section of the Review Queue.

**Categorisation Rules** are the dictionary that maps narration patterns to account categories. System rules (shipped with Ledger) and user rules (created from corrections or manually) are both stored here. User rules always take priority over system rules. Every rule records how many transactions it has matched, which helps the user gauge its effectiveness.

**Price History** records are associated with individual securities in investment accounts. Each record captures a price and a date. The most recent price record for any given security is used to compute the current market value in the Holdings tab.

**Reconciliation Periods** are attached to individual accounts. Each completed reconciliation records the statement balance, the target date, the transactions marked as cleared, and the user who completed it. Cleared transactions within a reconciliation period carry a lock flag that prevents silent modification.

If no dependencies exist, the undo proceeds:
1. Creates reversal journal entries for every JE in the batch (same date as original or
   today, configurable)
2. Updates all affected account running balances
3. Marks the import batch as "Rolled Back"
4. The transactions are removed from the Transaction List (archived, not deleted — visible
   in the import detail view under "Rolled Back Transactions")

### 7.3 Re-Process an Import

If a parser was updated to fix a bug (e.g., a new HDFC statement format change was handled
by a parser update), the user can re-process a previously imported file:

"Re-Process" re-runs the parser on the original stored file bytes. It first undoes the
original import (creating reversals), then processes the file afresh. The result lands in
the Review Queue as a new batch. This ensures the user gets the benefit of parser improvements
without having to find and re-upload old files.

## Sub-module 8: Statement Reconciliation

**Purpose:** Verify that the Ledger's balances for a given account exactly match the official statement from the bank or financial institution, catching any discrepancies due to missing transactions, duplicate entries, or data entry errors.

**Primary persona:** The methodical user — someone who wants to be certain their Ledger is a perfect mirror of reality, not just an approximation. Reconciliation is optional, but for users who combine imports with manual entry, it is the only way to catch errors that accumulate over time.

**UX Concept — Closing the Books**

In corporate accounting, reconciling a bank account is a routine month-end task: match every ledger entry against the bank statement, identify any outstanding items, and confirm that the balances agree. For individuals, this level of rigour is unusual — but remarkably valuable. A reconciled account balance is a verified balance. An unreconciled balance is an estimate.

Ledger makes reconciliation accessible without making it mandatory. The Reconcile screen is always available from any account's ledger view, but it never nags. For users who want the additional assurance, it provides a clean, guided workflow that takes five to ten minutes per account per month. The visual centrepiece is a live **Difference counter** that runs in a sticky header as the user marks transactions as cleared — it ticks down toward zero, and the moment it reaches zero the screen responds with a green confirmation. That single moment of closure is the emotional reward that makes reconciliation habit-forming.

Statement Reconciliation is the process of matching every transaction in Ledger against the
transactions in an official bank or financial statement, to confirm that the Ledger is a
perfect mirror of reality. This is the personal finance equivalent of a bank reconciliation
in corporate accounting.

### 8.1 When Reconciliation is Needed

Reconciliation is most useful for bank accounts where the user both imports statements
(for the bulk of transactions) and enters manual transactions (for cash withdrawals and
payments that don't appear on statements). Over time, discrepancies can accumulate — a
missed cash transaction here, a double-recorded entry there. Reconciliation catches these.

### 8.2 Reconciliation Workflow

The Reconciliation screen is accessible from any account's ledger view via an "Reconcile"
button in the account header.

**Step 1 — Set the reconciliation target:**
The user enters the closing balance from their official statement as of a specific date:
"My HDFC Savings statement balance as of 31-Mar-2026 was ₹1,42,830."

**Step 2 — Mark transactions as cleared:**
The screen shows every unreconciled transaction in Ledger for this account up to the
target date. The user marks each transaction as "Cleared" (it appeared on the official
statement) or leaves it unchecked (it has not yet appeared on any statement — outstanding
transactions like recent UPI credits that haven't settled).

As the user marks transactions, a running "Reconciled Balance" counter at the top of the
screen updates in real-time:

Statement Balance: ₹1,42,830
Cleared Transactions: ₹1,38,450
Outstanding (5 items): +₹ 4,380
Ledger Balance: ₹1,42,830
Difference: ₹ 0 ✓


**Step 3 — Resolve differences:**
If the Difference is not zero, the system helps identify likely causes:
- "There are 3 transactions in Ledger with no match in your statement — these may be
  duplicates or manually entered errors." (Links to those transactions)
- "Your statement has a closing balance ₹500 higher than expected — this could be an
  unrecorded transaction."

The user resolves differences by adding missing transactions, removing duplicates, or
adjusting incorrect amounts. The reconciliation is saved when the Difference reaches zero.

**Step 4 — Lock reconciled transactions:**
Once the Difference is zero, the user clicks "Complete Reconciliation." All cleared
transactions are marked as "Reconciled" and receive a lock icon in the Transaction List —
they cannot be edited or deleted without first unlocking the reconciliation period.

This creates a clean, auditable "as-of" checkpoint for the account.

## Sub-module 9: Investment Account Specifics

**Purpose:** Provide additional tracking capabilities for quantity-based investment accounts — equities, mutual funds, ETFs, gold — where tracking money in and money out is not enough. The quantity held, average cost per unit, and current market value are equally important.

**Primary persona:** Any user who has stocks, mutual funds, or ETFs. This includes Persona 1 (Riya, who has Zerodha and Kuvera) and Persona 3 (Ananya, who just opened a Zerodha account). Persona 2 (Suresh) benefits for his SGBs and physical gold holdings.

**UX Concept — Your Portfolio Is More Than a Balance**

A bank account has one number: the balance. An investment account has many: what you put in, what it is worth now, how many units you hold, what each unit cost you on average, and how much you have gained or lost both in absolute terms and as an annualised return. The standard transaction list cannot capture this richness — it would reduce a mutual fund folio to a series of debits and credits that mean nothing in isolation.

The investment account view adds a **Holdings** layer on top of the standard transaction layer. The same account, two lenses: the accounting view (what happened) and the portfolio view (what you have and what it is worth). Users switch between them with a tab. Neither replaces the other — both are needed for a complete picture.

Investment accounts (equities, mutual funds, ETFs, gold) have additional capabilities
beyond the standard transaction list due to their quantity-tracked nature.

### 9.1 Holdings View

For any investment account, the account ledger view has a "Holdings" tab alongside the
"Transactions" tab. The Holdings tab shows:

| Column | Description |
|---|---|
| Security / Fund | Name with exchange/AMC |
| ISIN | International security identifier |
| Quantity | Current units/shares held |
| Avg. Cost | Volume-weighted average purchase price per unit |
| Cost Value | Quantity × Avg. Cost |
| Current Price | Latest market price |
| Market Value | Quantity × Current Price |
| Unrealised P&L | Market Value − Cost Value (with % change) |
| XIRR | Annualised return accounting for all cash flows |

Each row expands to show the individual purchase lots (for FIFO tracking):

| Buy Date | Qty | Buy Price | Current Value | Holding Period | Gain/Loss | Classification |
|---|---|---|---|---|---|---|
| 12 Jan 2023 | 50 | ₹2,340 | ₹3,12,000 | 2 yr 2 mo | +₹1,29,950 | LTCG |
| 08 Sep 2024 | 30 | ₹2,890 | ₹1,87,200 | 6 mo | +₹ 3,500 | STCG |

This lot-level view is the basis for tax-loss harvesting recommendations.

### 9.2 Price Updates

For market-linked holdings, a "Update Prices" button in the Holdings view fetches the latest
prices from the configured market data integrations (AMFI for mutual fund NAVs, NSE/BSE for
equities). Prices are stored as price-history records against each security, and the portfolio
market value is recomputed.

If market data APIs are not configured, the user can manually enter the latest price for any
security. A "Price as of [date]" label shows the staleness of the last price update.

### 9.3 Quantity-Tracked Transaction Entry

When a BUY transaction is entered manually for an investment account, the Full Entry form
adds two additional fields: Quantity (units/shares purchased) and Price per unit. Amount is
auto-computed as Quantity × Price. The system validates: Quantity × Price = Debit Amount.

For SELL transactions, a quantity picker shows available lots (with their dates and
quantities) and the user selects which lots to sell (defaulting to FIFO if not specified).
The capital gains classification (STCG/LTCG) is computed and shown before the user confirms.

## Part 10: Accounting Engine Constraints

The Transaction Manager operates on top of the double-entry accounting engine. The following
rules are enforced at the engine level and cannot be bypassed from the UI.

### 10.1 The Balancing Rule (R3.1)

Every journal entry must balance: total debits = total credits. Entries that do not balance
are rejected by the engine before being written to the database. The UI prevents the user
from submitting an unbalanced entry (the "Save" button is disabled until the entry balances).

### 10.2 Immutability (R3.6)

Confirmed entries cannot be edited. The system achieves correction-through-reversal
transparently (see §3.5). The audit trail is immutable — even system administrators cannot
alter historical entries, only add reversals.

### 10.3 Account Type Constraints (R3.2)

The engine enforces normal balances:
- Assets and Expenses are increased by debits and decreased by credits.
- Liabilities, Equity, and Income are increased by credits and decreased by debits.

If a user attempts to manually create an entry that would cause an asset account to have
a credit balance (negative asset) without a corresponding contra-asset setup, the system
warns: "This entry creates a negative balance in [asset account]. Is this intentional?"
The user can override the warning.

### 10.4 Period Boundaries (R3.5)

The system supports both calendar year (Jan–Dec) and Indian financial year (Apr–Mar) as
accounting periods. The user's preferred period is set in Settings. All period-based
views in the Transaction List and all reports respect the selected period definition.

Entries cannot be dated before the account's opening date (the date of the first opening
balance entry for that account). Attempting to do so shows an error: "This date is before
the opening balance date for [account name]."
